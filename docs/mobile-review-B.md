# 모바일 디자인 리뷰 B — 결정 + 체크리스트 (2026-06-25)

> 새 핸드오프(design_handoff_A) + 스크린샷 3장 리뷰. 축: **[디자인]**(핸드오프/프로토타입) · **[구현]**(Compose 앱, 시연 대상).
> 용어 통일: SO=**재고이동요청** · IN_FULFILLMENT=**도착 대기** · RECEIVED=**입고 완료**.

## 사실 확인 (질문 답)
- **매핑대기**: 백엔드 상태 아님. user `/users/me`는 `tenancyName`(지점/창고명)만, `warehouseCode` 없음 → inventory 창고목록 이름매칭(PR#18)으로 해결. → 마이의 "매핑대기/연동 준비중"은 가상 상태라 **제거**.
- **알림**: `Notification.targetRole` 버킷(HQ_MANAGER / 지점 창고명)뿐. **개인 알림 없음** = 지점 공유 1종. `GET /notifications`가 지점 호출자에 지점 알림 반환(PR#84). → "내 알림 vs 지점 알림" 구분 폐기, "준비중"도 실제론 연동 가능.
- **용어**: IN_FULFILLMENT="도착 대기"(모바일), "처리중(충당)"(웹). **"입고 대기"는 오용** → 도착 대기로.
- **지점 이력 쿼리**: `GET /api/v1/sales-orders` 지점스코프 + status·requested_by·received_by·기간 + 응답에 requestedBy/approvedBy/receivedBy/canceledBy 모두 있음.

## 재고 조회 (Inventory)
- [ ] [구현] **상태(정상/부족/없음) ↔ 카테고리 필터 분리** — 현재 Compose는 한 Row에 통합(Inventory.kt FilterChips). 상태=세그먼트(또는 윗줄), 카테고리=아랫줄 칩으로 분리. (핸드오프는 이미 분리 의도.)
- [ ] [구현] **부품상세 "이 부품 입고 스캔" 버튼 제거** — 우리는 SO 전량 일괄 스캔만, SKU 단위 입고 없음 → 라벨이 모델과 불일치. (Inventory.kt PartSheet 378-392)
- [ ] [구현] **부품상세 1차 액션 = "이 품목 출고"만 유지** — 현장 동선(재고 보고 바로 출고). "재고이동요청"(웹 안내 버튼)은 가치 낮아 제외, STR은 웹.
- [ ] [구현/디자인] **"최근 입고"를 "도착 예정"으로 재정의 + 2~3건** — 사용자 인사이트("부족하지만 도착 예정 있으니 더 안 시켜도 됨")엔 RECEIVED(이미 입고)가 아니라 **이 부품 포함 IN_FULFILLMENT(도착 대기)**가 핵심. 라벨 "도착 예정", 상위 3건. (입고완료 이력은 작업이력 탭.)
